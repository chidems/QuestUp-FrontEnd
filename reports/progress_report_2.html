Current implementation

The defining event of this period was switching the app from its built-in mock API over to the real backend — a Dockerized FastAPI service backed by Postgres and Redis. That single change did more to reveal the true state of the project than any amount of testing against fixtures: several screens that looked finished in mock mode turned out to depend on backend data or schema fields that didn't exist yet. Most of this period's work went into finding and closing those gaps, alongside the planned onboarding and avatar-fit work — plus, in its final days, a null-response crash on the Events tab and a close look at why quest difficulty still reads wrong for every quest.

Friction log

Challenges encountered

Clothing didn't fit the body
The chibi avatar body and its 465 clothing sprites were drawn at different proportions, so garments were either stretched (oversized) or let skin show through at the shoulders and legs. Diagnosed and tuned with an offline compositing script rather than by rebuilding the app on every pass — see Stack notes.
A full rebuild, reverted
Tried solving the fit problem generally by generating an entirely new wardrobe from scratch. The result was structurally correct but visually flatter than the original hand-illustrated art, so it was reverted at review in favor of fixing the original sprites' fit numerically and keeping their art.
Mock data hid real gaps
Once pointed at the real backend, the Map screen showed 0 quests nearby and the Shop showed 7 bare items. Both traced to the backend's response schema and seed data, not the frontend: quest coordinates existed in the database but were never included in the API response, and the item catalog was seeded with 7 rows against a 91-item frontend asset set.
No backend write access
Cross-repo fixes had to be handed off rather than made directly: change-request documents were written with the exact schema, seed-data, and logic edits needed, for the backend owner to apply — then the frontend was updated once those landed.
A test was wrong, not the code
A long-failing unit test (NPCEncounter.fromJson) turned out to be asserting against a response shape the real backend never sends. Traced the parser against the actual backend schema to confirm the app was correct, then rewrote the test's fixture.
The emulator doesn't know where it is
Android emulators report a fixed Mountain View, CA GPS fix with no real hardware behind it. Verifying the map/location work required manually pushing a coordinate fix (adb emu geo fix) to simulate a real position for testing and demo purposes.
A null response was a valid state, not an error
GET /community/weekly/current legitimately returns 200 null when no weekly community quest is active between cycles, but the Events screen cast that response straight to a non-null map, so it crashed to the error view every time instead of showing an empty state. Made null a handled case end-to-end — API client, repository, provider, and screen — instead of special-casing it at one call site.
A backend fix landed for the wrong bug
A backend commit arrived claiming to fix the reported difficulty issue, but only touched the weekly-quest rollover — a real, separate bug this period also surfaced. The quest-difficulty override documented earlier was untouched. Caught by reading the live Postgres database and git log directly rather than trusting the commit message, and re-flagged with the exact lines still unpatched.
Stack notes

New techniques & tools applied

Offline sprite compositor
A Python/NumPy/Pillow script that mirrors the Flutter app's exact layer and anchoring math, used to iterate on avatar clothing fit dozens of times without a single app rebuild — only the final result was verified on-device.
Procedural pixel art
Generated flat-color, outlined pixel-art icons programmatically, for a shop-icon stopgap ahead of the real catalog fix.
Dockerized backend integration
Ran the app against a real FastAPI + PostgreSQL + Redis stack in Docker, replacing the mock-mode flag used through earlier phases.
Cross-repo schema debugging
Diagnosed frontend symptoms by reading the backend's Python/SQLAlchemy models and Pydantic response schemas directly, rather than guessing from API responses alone.
Direct container/DB verification
Used docker exec into the backend's Postgres and API containers to read live table contents and confirm which code version was actually running, rather than inferring backend state from API responses alone — this is how the still-open difficulty bug and the working rollover fix were both confirmed with certainty.
adb-driven verification
Used Android Debug Bridge directly for repeatable on-device checks — screenshotting, uiautomator-dumped element bounds for precise tap automation, force-stopping/relaunching, and injecting a GPS fix — so each fix (and this report's screenshots) could be confirmed against the actual running app.

Scope delta

Changes to timeline & scope

MVP status was optimistic. CLAUDE.md described all 7 build phases as complete with only minor polish remaining. Pointing the app at the real backend showed that several "done" features — map pins, the store catalog's depth — were done against mock data only, and needed backend-side fixes once real data was involved. That work absorbed most of this period.
Onboarding changed shape twice. First built as a direct preference-selection form, then rebuilt as an inferential four-question quiz after review — same backend fields, a different, less on-the-nose way of asking for them.
Avatar art direction resolved. The Stardew Valley–styled concept explored last period was reviewed and not adopted — the original hand-illustrated wardrobe, with this period's fit corrections, stays in production. No new art scope carries forward from that exploration.
Cross-repo bug attribution mattered twice this period. A reported backend fix didn't cover what its commit message claimed — verifying against live data and git history, not just the message, was the only way to catch that a "quest difficulty" fix request had actually landed a different, real fix (weekly-quest rollover) instead.
