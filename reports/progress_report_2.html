Most of this period's work went into finding and closing those gaps, alongside the planned onboarding and avatar-fit work.

Friction log

Challenges encountered

Clothing didn't fit the body
The chibi avatar body and its 465 clothing sprites were drawn at different proportions, so garments were either stretched (oversized) or let skin show through at the shoulders and legs. Diagnosed and tuned with an offline compositing script rather than by rebuilding the app on every pass — see Stack notes.
A full rebuild, reverted
Tried solving the fit problem generally by generating an entirely new wardrobe from scratch. The result was structurally correct but visually flatter than the original hand-illustrated art, so it was reverted at review in favor of fixing the original sprites' fit numerically and keeping their art.
Mock data hid real gaps
Once pointed at the real backend, the Map screen showed 0 quests nearby and the Shop showed 7 bare items. Both traced to the backend's response schema and seed data, not the frontend: quest coordinates existed in the database but were never included in the API response, and the item catalog was seeded with 7 rows against a 91-item frontend asset set.
No backend write access
Cross-repo fixes had to be handed off rather than made directly: two change-request documents were written with the exact schema and seed-data edits needed, for the backend owner to apply — then the frontend was updated once those landed.
A test was wrong, not the code
A long-failing unit test (NPCEncounter.fromJson) turned out to be asserting against a response shape the real backend never sends. Traced the parser against the actual backend schema to confirm the app was correct, then rewrote the test's fixture.
The emulator doesn't know where it is
Android emulators report a fixed Mountain View, CA GPS fix with no real hardware behind it. Verifying the map/location work required manually pushing a coordinate fix (adb emu geo fix) to simulate a real position for testing and demo purposes.
Stack notes

New techniques & tools applied

Offline sprite compositor
A Python/NumPy/Pillow script that mirrors the Flutter app's exact layer and anchoring math, used to iterate on avatar clothing fit dozens of times without a single app rebuild — only the final result was verified on-device.
Procedural pixel art
Generated flat-color, outlined pixel-art icons and a full alternate wardrobe programmatically, for both a shop-icon stopgap and the new avatar concept exploration.
Dockerized backend integration
Ran the app against a real FastAPI + PostgreSQL + Redis stack in Docker for the first time this period, replacing the mock-mode flag used through earlier phases.
Cross-repo schema debugging
Diagnosed frontend symptoms by reading the backend's Python/SQLAlchemy models and Pydantic response schemas directly, rather than guessing from API responses alone.
adb-driven verification
Used Android Debug Bridge directly for repeatable on-device checks — screenshotting, simulating taps, force-stopping/relaunching, and injecting a GPS fix — so each fix could be confirmed against the actual running app.

Scope delta

Changes to timeline & scope

MVP status was optimistic. CLAUDE.md described all 7 build phases as complete with only minor polish remaining. Pointing the app at the real backend showed that several "done" features — map pins, the store catalog's depth — were done against mock data only, and needed backend-side fixes once real data was involved. That work absorbed most of this period.
Onboarding changed shape twice. First built as a direct preference-selection form, then rebuilt as an inferential four-question quiz after review — same backend fields, a different, less on-the-nose way of asking for them.
A new scope item opened. The avatar wardrobe rebuild was fully built, then reverted at review in favor of a fit fix to the original art. A new avatar art direction was explored in its place and is now a pending decision rather than committed scope.
